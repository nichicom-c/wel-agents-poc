# law RAG Hierarchical chunking 比較 最終報告

確認日: 2026-06-19

## Summary

同一 query に対して、現行 `law` Knowledge Base（S3 Vectors）と `law_hierarchical` Knowledge Base（OpenSearch Serverless + `HIERARCHICAL` chunking）を `Retrieve` で比較した。今回の query では検索順位とスコアはほぼ同等だが、Hierarchical 版は同じ上位文書をより広い条文単位の文脈で返しており、回答生成時の根拠の完全性は Hierarchical 版がやや優位だった。

結論として、今回のデータだけで「検索ランキング精度が上がった」とは言えない。一方で、法令 RAG のように同一条文内の複数項を正確に含める必要がある用途では、Hierarchical 版は回答の根拠落ちを減らす可能性がある。

## Evaluation Input

Query:

```text
児童虐待を受けたと思われる児童を発見した場合、誰にどのような通告義務がありますか？根拠条文も示してください。
```

Retrieve settings:

| 項目 | 値 |
| --- | --- |
| numberOfResults | 5 |
| current | 現行 `law` Knowledge Base（S3 Vectors） |
| hierarchical | `law_hierarchical` Knowledge Base（OpenSearch Serverless + `HIERARCHICAL` chunking） |

## Retrieve Result Comparison

| 観点 | current | hierarchical | 評価 |
| --- | --- | --- | --- |
| 上位文書の順位 | 第六条、第八条、第七条、第十三条の五、第八条の二 | 第六条、第八条、第七条、第十三条の五、第八条の二 | 同順位で差なし |
| top score | 0.9270239770412445 | 0.9270095 | 誤差レベル |
| 取得本文量 | 1,074文字 | 2,376文字 | hierarchical は約2.2倍 |
| 第六条の取得範囲 | 第1項、第2項 | 第1項、第2項、第3項 | hierarchical が優位 |
| 第八条の取得範囲 | 冒頭のみ | 第1項各号、第2項各号の一部まで | hierarchical が優位 |
| ノイズ / token 効率 | 短く、回答に必要な主根拠へ集中 | 文脈が厚い分、不要文脈と token は増える | current が優位 |

今回の query で最も重要な根拠は、児童虐待防止法第六条第一項である。両者とも第六条を top で取得できているため、主回答の骨子はどちらでも作れる。

差分として、Hierarchical 版は第六条第三項まで含んでいる。第六条第三項は「秘密漏示罪その他の守秘義務規定は、通告義務の遵守を妨げるものと解釈してはならない」という補足であり、専門職や守秘義務との関係を説明する場合に有用である。

## Assessment

| 評価軸 | 判定 |
| --- | --- |
| 検索ランキング | 引き分け。top 5 の文書順序は完全一致した。 |
| 主根拠の取得 | 引き分け。どちらも第六条を top で取得した。 |
| 根拠の完全性 | Hierarchical がやや優位。第六条第三項まで含めて返せている。 |
| 回答の簡潔性 | current が優位。取得本文が短く、主回答に集中しやすい。 |
| 法令 RAG への適性 | Hierarchical がやや優位。条文内の複数項や周辺規定を含めやすい。 |

今回の結果では、Hierarchical 版は「関連文書を新たに見つける」方向ではなく、「同じ関連文書をより完全な文脈として返す」方向で効いている。これは法令・規程・マニュアルのように見出し、条、項、号の階層構造が回答品質に影響する corpus では有効な改善である。

ただし、取得本文量が増えるため、常に Hierarchical 版へ切り替える判断はまだ早い。複数 query で、回答の根拠漏れ、引用条文の正確性、不要文脈による回答のぶれ、token cost を比較する必要がある。

## Recommendation

短期的には、通常 Chat UI の `law_rag_agent` は現行 `LAW_KB_ID` のまま維持し、`law:compare:hierarchical` CLI で評価 query を増やして比較を継続する。

評価 query は、単純な条文特定だけでなく、次のような階層文脈が必要なものを含める。

- 同一条文の複数項をまたぐ質問
- 条文と関連条文の関係を説明する質問
- 通告後の措置、守秘義務、出頭要求など、手続きの流れを問う質問
- 「誰が」「どこへ」「いつ」「例外はあるか」を同時に聞く質問

Chat UI 上で比較したい場合は、別途 UI / BFF / AgentCore に比較モードを追加する必要がある。現状の実装は、通常チャットで比較表示するものではなく、CLI で同一 query の Retrieve 結果を並べる評価用実装である。

## Next Steps

1. 5〜10個の評価 query を用意し、`current` と `hierarchical` の Retrieve JSON を保存する。
2. 各 query について、top result、必要条文の有無、取得本文量、回答例の根拠漏れを同じ表で採点する。
3. Hierarchical 版が一貫して根拠漏れを減らす場合、`law_rag_agent` の切り替え方針を検討する。
4. UI 比較が必要な場合は、Chat UI ではなく Dev / Evaluation 画面または CLI report として追加するのがよい。

## Decision

現時点の判定は、`law_hierarchical` は有効な比較対象として動作しているが、通常回答経路へ即時切り替える根拠としては不足である。法令 RAG の精度改善候補として、追加評価を継続する。
